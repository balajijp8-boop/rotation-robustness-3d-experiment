\section{Motivation and Hypothesis}

Automatic 3D object classification is essential for autonomous driving and robotic manipulation, where LiDAR sensors produce \textbf{point clouds} (unordered sets of 3D coordinates). A key practical challenge: sensors don't guarantee that objects appear in a canonical orientation. A robust classifier must recognise objects regardless of pose.

Point-based methods operate directly on raw 3D coordinates.
\href{https://arxiv.org/abs/1612.00593}{PointNet (Qi et al., 2017)} applies shared MLPs per point and pools a global descriptor. It includes a \emph{T-Net}, a sub-network that predicts a $3\times3$ transformation matrix and applies it to the input before classification, giving the network an explicit handle on rotation.
\href{https://arxiv.org/abs/1801.07829}{DGCNN (Wang et al., 2019)} constructs a local $k$-NN graph and learns edge features between neighbours.

Projection-based methods convert point clouds to 2D images first.
\href{https://arxiv.org/abs/2106.05304}{SimpleView (Goyal et al., 2021)} renders each point cloud as \textbf{six orthogonal depth images} ($\pm X, \pm Y, \pm Z$ at $128\times128$), processes each with a compact ResNet-18, and classifies the concatenation of the six feature vectors.

On ModelNet40, SimpleView matches or outperforms point-based methods with fewer parameters. On \textbf{ScanObjectNN}, a real-world dataset where objects are \emph{not} axis-aligned and are surrounded by background clutter, SimpleView achieves \textbf{79.5\% accuracy}, outperforming DGCNN (78.1\%).

\medskip
\noindent\textbf{But ScanObjectNN confounds multiple variables at once.} Objects aren't axis-aligned, they're partially occluded, and there is background clutter everywhere. It is impossible to tell from these results alone whether SimpleView's advantage comes from rotation robustness, noise and occlusion robustness, or some other factor.

\medskip
This distinction matters because the two families handle rotation \emph{fundamentally differently}. SimpleView projects to six \emph{fixed world-space} directions, so if an object rotates, each depth image changes because a different face is visible from each fixed camera. PointNet's T-Net, by contrast, attempts to explicitly canonicalise the input before processing.

\begin{hypothesisbox}
\textbf{Hypothesis:} If SimpleView's advantage on ScanObjectNN comes from rotation robustness, it should degrade \emph{more gracefully} than PointNet and DGCNN when objects are rotated at test time; seeing all six faces simultaneously should provide complementary information at intermediate angles.

\medskip
\textbf{This is directly testable} with a fully controlled experiment where rotation is the \emph{only} variable (van Gemert RP9: change one variable at a time).
\end{hypothesisbox}

\section{Dataset Description and Examples}

To isolate rotation as the single controlled variable, we generate a synthetic dataset of \textbf{five geometrically distinct 3D shape classes}:

\begin{center}
\begin{tabular}{ll}
\toprule
\textbf{Class} & \textbf{Sampling method} \\
\midrule
Cube     & Uniform over all six faces \\
Sphere   & Normalised Gaussian vectors (exactly uniform surface) \\
Cylinder & Area-proportional: lateral surface $+$ two circular caps \\
Cone     & Area-proportional: lateral surface $+$ base disk \\
Torus    & Rejection sampling, weight $(R + r\cos\theta)/(R+r)$ \\
\bottomrule
\end{tabular}
\end{center}

Every point cloud contains exactly \textbf{1,024 points}, normalised to zero-mean and unit maximum radius.

\medskip
\noindent\textbf{What the dataset does NOT include:} no noise, no background clutter, no occlusion, no class imbalance, no difference in point density. The only thing that changes between training and any test condition is the \textbf{rotation angle applied to each point cloud}. This strict isolation is what makes it a valid control experiment.

\section{Method of Generation}

All generation code is scripted end-to-end for full reproducibility (van Gemert RP11). No manual steps.

\subsection*{Shape Samplers}

Each class uses a custom, area-correct sampler to ensure uniform surface coverage:

\begin{itemize}[leftmargin=2em]

\item \textbf{Cube}: Points are drawn uniformly on each of the 6 faces with per-face counts proportional to face area (all equal for a unit cube).

\item \textbf{Sphere}: Sample from $\mathcal{N}(0, I_3)$ and normalise each vector to unit length. The normalisation step is what produces exactly uniform surface coverage, not the Gaussian itself.

\item \textbf{Cylinder} (radius$=1$, height$=2$): Lateral area $= 4\pi$, each cap area $= \pi$, giving a 4:1:1 allocation. Lateral points use uniform $\theta \in [0,2\pi]$ and uniform $z \in [-1,1]$; cap points use $r = \sqrt{u},\ u \sim \mathrm{Uniform}[0,1]$ to correct for the area compression at the centre of a disk.

\item \textbf{Cone} (apex at $z=+1$, base radius$=1$, slant $= \sqrt{5}$): Uniform placement on the curved lateral surface requires $r \sim \sqrt{u}$ rather than $r \sim u$, otherwise points cluster near the apex where surface density is lower.

\item \textbf{Torus} ($R=0.65$, $r=0.25$): Rejection sampling: draw $(\theta, \phi) \sim \mathrm{Uniform}[0,2\pi]^2$ and accept with probability $(R+r\cos\theta)/(R+r)$. The outer rim of the torus subtends more arc length per radian, so without this weight points would oversample the inner rim.

\end{itemize}

\subsection*{Normalisation}

Every point cloud is zero-centred (subtract mean) and scaled to the unit sphere (divide by max radius). All classes occupy the same spatial scale.

\subsection*{Rotation}

\begin{itemize}[noitemsep, leftmargin=2em]
  \item \textbf{Y-axis:} $3\times3$ rotation matrix $R_y(\theta)$ applied to all points.
  \item \textbf{SO(3):} \texttt{scipy.spatial.transform.Rotation.random()} for uniformly distributed random orientations. Three independent rotation seeds for stability.
\end{itemize}

\subsection*{The Critical Control}

\begin{findingbox}
\textbf{No rotation augmentation is applied during training.} All three models see only canonical-orientation shapes. Any accuracy difference under test-time rotation is attributable entirely to the \emph{architecture}, not the training protocol, which is exactly what RP9 requires.
\end{findingbox}

\subsection*{Results (van Gemert c1--c4)}

Every model reaches 100\% at $0^\circ$. The shapes are geometrically distinguishable and no architecture enters the rotation tests at a disadvantage.

Rotation hurts, as expected. All three methods fall to around 60\% between $90^\circ$ and $120^\circ$, well clear of the 20\% random floor but far enough below aligned performance to confirm the effect is real.

SimpleView at 60.3\% SO(3) is not just below PointNet; it sits at the dataset floor. That is a strong result: the multi-view architecture gives no measurable lift over pure geometric symmetry. PointNet's T-Net, the mechanism we expected to lose to six-camera coverage, marginally outperforms both.

\subsection*{Why Accuracy Floors at 60\%}

Three of the five shapes are symmetric under Y-axis rotation: the \textbf{sphere} (fully symmetric), the \textbf{torus} (Y-symmetric), and the \textbf{cube} (all faces are identical squares). These are classified correctly regardless of angle. The \textbf{cylinder} and \textbf{cone} have a preferred axis; their appearance changes substantially at intermediate angles and all models fail on both. The floor is $3/5 = 60\%$ and follows directly from the geometry of the dataset.

\subsection*{Why SimpleView Does Not Help}

SimpleView's six cameras point in fixed world-space directions ($\pm X, \pm Y, \pm Z$). When a cylinder rotates $90^\circ$ around Y, the $+Z$ view now shows the cylinder cap instead of its lateral surface, a view pattern never seen during aligned training. ResNet-18 has no mechanism to generalise from this, and crucially there is \textbf{no pooling over view directions}: each view contributes a separate feature vector that is concatenated. An unseen view produces an unseen feature with no learned invariance to compensate.

\subsection*{The V-Shaped Accuracy Curve}

Accuracy recovers at $180^\circ$ because a half-turn around Y maps most shapes back to an appearance close to $0^\circ$: the cone apex still points up, the cylinder is still vertical. All five classes are classified correctly at $180^\circ$, producing the symmetric dip centred near $90^\circ$.

% ─────────────────────────────────────────────────────────────
\section{Conclusion}

SimpleView's six fixed cameras are, in a sense, too rigid: without global pose alignment, an unseen viewpoint simply breaks the feature concatenation. PointNet wins SO(3) at 66.7\% versus SimpleView's 60.3\%, which lands exactly at the dataset floor.

\begin{findingbox}
The multi-view architecture adds nothing here. This suggests that SimpleView's gain on ScanObjectNN likely comes from something other than rotation robustness: noise suppression, occlusion handling, or perhaps the implicit smoothing that depth projection applies to background clutter. Those are all factors this experiment deliberately excluded. A follow-up varying clutter density with pose held fixed would test that directly (van Gemert RP9).
\end{findingbox}

\subsection*{Limitations}

\begin{itemize}[noitemsep, leftmargin=2em]
  \item Only five simple geometric primitives; real-world objects may yield different conclusions.
  \item 50 epochs without rotation augmentation; augmented training might change the relative ranking.
  \item Each model was trained once (single training seed); the 3 seeds in the SO(3) condition vary only the test-time rotations. Repeated training runs would reduce variance further.
\end{itemize}
